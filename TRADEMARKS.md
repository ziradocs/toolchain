# Trademark Policy

"SlideLang" and "DocLang", together with their logos (where used), are trademarks of the SlideLang
project ("we", "the project"). This policy explains how you can use these names and marks, and
how to talk about compatibility with the project without implying an affiliation or endorsement
that doesn't exist.

This document covers only the marks actually shipped from this repository: **SlideLang** (the
presentation DSL and `slidelang` CLI) and **DocLang** (the document DSL and `doclang` CLI). It
does not cover any other product name, commercial or otherwise — those are out of scope here and
governed separately if and when they exist.

The project's code is licensed under Apache License 2.0 (see [LICENSE](LICENSE)). **The Apache-2.0
license grants copyright and patent rights to the code — it does not grant trademark rights.**
Using the code under Apache-2.0 does not by itself give you permission to use the SlideLang or
DocLang names or logos beyond what's described below.

## What you can do without asking

- Use, modify, and redistribute the SlideLang/DocLang source code under the terms of the
  Apache-2.0 license.
- Refer to the project by name in prose, documentation, blog posts, talks, and comparisons
  ("built with SlideLang", "generates output similar to DocLang", "a DocLang parser written in
  Rust").
- State factually that your project is **compatible with** or **built on top of** SlideLang or
  DocLang, e.g. "SlideLang-compatible", "works with DocLang output" — as long as it's accurate and
  doesn't imply official endorsement.
- Use the unmodified project logo (once one exists) to link back to or reference the project, in
  the same way you'd cite any other open-source project you depend on.

## What requires permission

- Naming your own product, fork, service, or company "SlideLang" or "DocLang" (or a name
  confusingly similar to either), including as a prefix/suffix (e.g. "SlideLangPro",
  "DocLang Cloud") in a way that could be mistaken for an official project offering.
- Using the SlideLang or DocLang name/logo in a way that suggests your project is the official
  distribution, is endorsed by, or is affiliated with the SlideLang project, when it isn't.
- Modifying the logo (colors, proportions, combining with other marks) and continuing to present
  it as the SlideLang/DocLang mark.
- Using the marks in domain names, app store listings, or social media handles in a way that could
  be mistaken for an official project account.

If you want to do any of the above — for example, a distribution maintained by your organization,
or a hosted service built on the project that you'd like to name after it — open an issue or reach
out to the maintainers to discuss it. Most reasonable requests (accurate "powered by" branding,
clearly-labeled forks for a specific ecosystem, etc.) are fine; we just want to avoid user
confusion about what's official.

## Forks

You're free to fork this project under Apache-2.0. We just ask that a fork not called "SlideLang"
or "DocLang" doesn't present itself as such — pick a distinct name, and feel free to note that
it's "a fork of SlideLang" or "based on DocLang" for context.

## Registration status

SlideLang and DocLang are project names currently protected as unregistered ("common law") marks
through use. Trademark applications may be filed in the future; this policy will be updated to
reflect registration status if and when that happens. Nothing in this document should be read as
a representation that a registration currently exists.

## Organization

The project's source is hosted under the `ziradocs` GitHub organization
(`github.com/ziradocs/toolchain`), a neutral open-source steward organization, alongside DocLang,
as peer products under one non-product-specific home. `ziradocs` is the name of that steward
organization only; it is not itself a product name covered by this policy.

## Questions

If you're unsure whether a particular use is fine, ask first by opening a GitHub issue — it's
much easier to sort out before you've shipped something than after.

This policy is inspired by common patterns in other open-source trademark policies (e.g. Python
Software Foundation, Rust Foundation) adapted to this project's scale. It is not legal advice, and
the project may seek professional counsel to formalize it further as the project grows.
